# LLMTrio 기획서

## 한 줄 정의

프롬프트 하나 넣으면 Claude, Codex, Gemini가 동시에 일하고, 유저는 하나의 AI를 쓰는 것처럼 결과를 받는다.

## 왜 만드는가

LLM 구독 모델이 보편화됐다. Claude Max, Codex, Gemini — 개발자 한 명이 3개를 동시에 구독하는 경우가 늘고 있다. 하지만 실제로는 터미널을 3개 띄워놓고 하나씩 번갈아 쓴다. 복붙하고, 결과 비교하고, 어느 모델이 더 나은지 직접 판단한다.

이 과정을 자동화하면 어떨까. 프롬프트 하나를 넣으면 각 모델이 잘하는 일을 알아서 나눠 맡고, 결과를 합쳐서 돌려주는 도구. 유저는 "AI 하나를 쓰는 느낌"이지만 실제로는 3개가 병렬로 도는 구조.

이게 LLMTrio의 출발점이다.

## 누가 쓰는가

**1차 타겟:** AI 코딩 CLI를 이미 쓰는 1인 개발자. Claude Code, Codex, Gemini CLI 중 하나 이상 구독 중. 터미널에서 코드를 짜고, 더 나은 결과를 원한다.

**2차 타겟:** 소규모 개발팀. 팀원마다 다른 AI 도구를 쓰고 있고, 결과를 통일하고 싶다. 코드 리뷰, 문서화, 아키텍처 설계를 AI에게 분담시키고 싶다.

**공통 특성:** CLI에 익숙하지만, 진행 상황을 눈으로 보고 싶다. 결과를 비교하고 선택하고 싶다. "그냥 되는" 도구를 원한다.

## 핵심 가치 제안

### 유저가 느끼는 것
- 프롬프트 하나 입력
- 브라우저에서 진행 상황을 실시간으로 봄
- 어떤 AI가 무슨 작업 중인지 한눈에 파악
- 결과를 한 번에 받음
- 필요하면 비교하고 채택

### 실제 벌어지는 것
- 프롬프트가 태스크로 분해됨
- 각 태스크가 적합한 모델에 할당됨 (Claude=설계/리뷰, Codex=구현, Gemini=리서치/문서)
- 병렬로 동시 실행됨
- 결과가 합쳐져서 정리됨

### 경쟁 대비 차별점
- **GUI로 보는 오케스트레이션.** 다른 도구는 CLI 결과를 텍스트로 뿌린다. LLMTrio는 브라우저 대시보드에서 흐름도로 보여준다. 어떤 모델이 어떤 태스크를 맡았는지, 지금 어디까지 진행됐는지, 에러가 나면 어디서 났는지.
- **하나처럼 쓰지만 셋이 도는 구조.** 유저 입장에서는 하나의 인터페이스. 뒤에서는 3개 구독 모델이 각자 역할을 수행.
- **로컬 퍼스트.** 클라우드 없음. API 키가 외부로 나가지 않음. `npx llmtrio` 한 방에 localhost에서 실행.

## 핵심 기능

### 1. 워크플로우 엔진

프롬프트를 2단계로 분해해서 실행한다.

**Plan 단계** — 3개 모델이 동시에:
- Gemini: 요구사항 분석, 기술 리서치 (넓게 본다)
- Claude: 아키텍처 설계, 컴포넌트 구조 (깊이 생각한다)
- Codex: 프로젝트 골격 코드 생성 (빠르게 짠다)

**Execute 단계** — Plan 결과를 바탕으로 3개 모델이 동시에:
- Codex: 핵심 코드 구현
- Claude: 코드 리뷰 (버그, 보안, 개선점)
- Gemini: 문서 작성 (README, 사용법)

Plan이 전부 성공해야 Execute로 넘어간다. 같은 단계 안에서는 병렬 실행된다.

### 2. 승인 모드

Plan 결과를 유저가 검토한 뒤 Execute 진행 여부를 결정한다. "AI가 알아서 다 하는 것"과 "사람이 중간에 판단하는 것"을 선택할 수 있다.

- **승인 모드:** Plan 완료 → 유저 검토 → 승인하면 Execute → 결과
- **자동 모드:** Plan → Execute 바로 연결 → 결과

승인 모드에서 유저는 Plan 결과를 보고 "이 방향 맞다" 또는 "다시 해" 를 판단한다. Execute에는 Plan 결과가 context로 주입되므로 승인한 방향대로 구현이 진행된다.

### 3. 대시보드 (브라우저 GUI)

유저가 보는 유일한 인터페이스. 터미널 없이 브라우저만으로 모든 것을 한다.

**흐름도 (Flow):** Plan→Execute 구조를 시각화. 각 노드가 어떤 모델에서 어떤 태스크를 처리 중인지 실시간 표시. 완료 시 애니메이션. 노드 클릭하면 상세 결과.

**결과 (Results):** 모든 태스크 출력을 합쳐서 보여준다. AI 요약 자동 생성. 원본 출력 접기/펼치기.

**비교 (Comparison):** 에이전트별 결과를 나란히 비교. 어떤 모델의 결과가 더 나은지 직접 판단하고 채택.

**Setup:** CLI 설치, 인증, 워크플로우 설정을 대시보드에서 처리. 첫 사용자가 터미널 없이 시작할 수 있다.

**브리핑:** 워크플로우 완료 시 팝업으로 결과 요약, 실패 태스크 안내, 다음 액션 제안.

### 4. 워크플로우 템플릿

매번 6개 태스크를 다 돌릴 필요 없다. 용도에 맞는 프리셋을 제공한다.

- **Full (6 태스크):** 리서치 + 설계 + 스캐폴드 → 구현 + 리뷰 + 문서. 새 프로젝트 시작용.
- **Quick (2 태스크):** 설계 → 구현. 빠르게 코드만 필요할 때.
- **Review (2 태스크):** 리서치 → 코드 리뷰. 기존 코드 점검용.
- **Docs (2 태스크):** 리서치 → 문서 작성. README나 가이드 필요할 때.

### 5. 실패 복구

워크플로우 중 일부 태스크가 실패해도 전체를 다시 돌리지 않는다.

- 실패한 태스크만 골라서 재시도
- 성공한 태스크의 결과는 유지
- 에러 메시지에 액션 제안 ("타임아웃을 늘려보세요", "인증을 확인하세요")

### 6. 결과 채택

비교 후 마음에 드는 결과를 선택하면 파일로 저장된다. 현재는 `.trio/adopted/`에 마크다운 파일로 저장. 향후 git commit 연동 가능.

### 7. Repo-Aware Context

프롬프트만으로는 프로젝트 맥락을 전달하기 어렵다. LLMTrio는 현재 git 상태(branch, 최근 커밋, 변경 파일)를 자동으로 수집해서 에이전트 프롬프트에 주입한다. 유저가 별도로 컨텍스트를 설명할 필요가 줄어든다.

### 8. 세션 보안

localhost 도구이지만 기본적인 보안은 갖춘다. 서버 시작 시 세션 토큰을 생성하고, 모든 쓰기 API에 토큰 검증을 요구한다. API 키는 `.trio/.env`에 저장되고 에이전트 프로세스에는 전달되지 않는다.

## 현재 상태와 해결해야 할 문제

### 해결된 것
- 워크플로우 엔진 (plan→execute 병렬 실행)
- 승인 모드 / 자동 모드
- SSE 실시간 갱신
- 세션 토큰 보안
- 에러 복구 UX
- 워크플로우 템플릿
- 결과 비교 / 채택
- Repo-aware context
- E2E 테스트 16개
- 크로스 플랫폼 (macOS/Linux, Windows WSL)

### 해결해야 할 핵심 문제

**결과 품질이 하나의 LLM을 직접 쓰는 것보다 못하다.**

이게 가장 큰 문제다. 3개를 동시에 돌리는데 결과가 하나만 쓸 때보다 나빠지면 존재 이유가 없다.

원인 분석:
- 각 에이전트에게 가는 프롬프트가 원본 프롬프트의 일부만 전달된다. "코드 리뷰해줘"라는 간단한 지시만 받으면 맥락 없이 일반론적인 리뷰를 한다.
- Plan 결과가 Execute에 충분히 전달되지 않는다. 2000자로 잘린다.
- 에이전트가 프로젝트의 실제 코드를 보지 못한다. Repo context가 git status 수준이지 실제 파일 내용이 아니다.
- 결과를 합치는 과정이 단순 연결(concatenation)이다. 중복, 모순, 품질 차이를 정리하지 않는다.

해결 방향:
- 에이전트 프롬프트에 프로젝트의 실제 코드/파일을 포함시키기
- Plan→Execute context를 잘리지 않게 전달하기
- 결과를 합칠 때 AI가 정리/통합하는 단계 추가하기
- 각 태스크별 프롬프트를 더 구체적으로 작성하기 (현재도 구조화했지만 부족)

**진행 상황을 보기 어렵다.**

흐름도가 있지만, 태스크가 "working" 상태일 때 실제로 뭘 하고 있는지 알 수 없다. 완료된 후에야 결과가 보인다. 실시간 스트리밍이 없다.

해결 방향:
- 에이전트 stdout을 실시간으로 대시보드에 스트리밍
- "working" 상태의 노드를 클릭하면 현재까지의 출력을 볼 수 있게
- 경과 시간 외에 토큰 사용량, 출력 길이 등 진행 지표 표시

**UI가 기대에 못 미친다.**

디자인 시스템은 좋지만, 실제 사용 흐름에서 불편한 부분이 있다.
- 워크플로우 시작 시 피드백이 약하다 (로딩 스피너 추가했지만 충분하지 않다)
- 결과를 실제 프로젝트에 적용하는 마지막 단계가 복사/붙여넣기다
- 에러 상태에서 "다음에 뭘 해야 하는지"가 직관적이지 않다

## 기술 원칙

### 반드시 유지
- **Zero npm dependencies.** Node.js 빌트인만 사용. `npx llmtrio` 한 방에 동작.
- **로컬 퍼스트.** 클라우드 없음. 데이터가 외부로 나가지 않음.
- **GUI 중심.** 터미널 없이 브라우저만으로 설치, 설정, 실행, 결과 확인.
- **파일 기반 상태.** `.trio/` 디렉토리의 JSON 파일이 전부. DB 없음.

### 바꿔도 되는 것
- 싱글 파일 HTML → 이미 CSS/JS 분리함. 더 분리해도 됨.
- bash 스크립트 → Node.js로 대체 진행 중.
- 특정 에이전트 고정 → 에이전트 할당을 유저가 바꿀 수 있게.

```
│  └─────────┘ └─────────┘ └─────────┘   │
└──────────────────────────────────────────┘
                 │
        .trio/ (JSON files)
        ├── state.json        ← 워크플로우 상태

        ├── budget.json       ← 워크플로우 설정
```


## 수익화 전략

**Phase 1: 오픈소스 (현재)**
- MIT 라이선스
- npm 배포, GitHub 공개
- 커뮤니티 피드백 수집
- 블로그/SNS로 인지도 구축

**Phase 2: 프리미엄 (결과 품질 확보 후)**
- 코어는 오픈소스 유지
- 프리미엄 기능: 결과 통합 AI, 프로젝트 코드 분석, 팀 공유 프리셋
- 가격: 월 $9~19 (개인), 월 $29~49 (팀)

**Phase 3: SaaS (수요 확인 후)**
- 클라우드 버전 (API 키 관리, 팀 협업, 히스토리)
- 로컬 버전은 계속 무료 유지

## 포기한 기능과 이유

**Debate (에이전트 간 토론):** 구현 비용 대비 실제 효용이 낮았다. 에이전트끼리 대화시키면 토큰만 쓰고 결과 품질은 올라가지 않았다. 실시간 토론 UI도 만들었지만 백엔드가 없어서 유저를 속이는 꼴이었다. 제거.

**칸반 보드 (드래그앤드롭):** 보기에는 좋지만 실제로 태스크를 사람이 옮길 이유가 없었다. 워크플로우 엔진이 자동으로 관리하는 게 맞다. CSS만 남고 JS가 구현 안 된 상태였다. 제거.

**실시간 비용 추적:** 구독 모델에서는 건당 비용 개념이 없다. API 키 기반이면 의미 있지만, 구독이면 "이번 달 남은 사용량"이 더 중요하다. 근데 그건 각 CLI가 제공하는 정보이므로 LLMTrio가 관여할 수 없다. 제거.

**자동 모델 추천:** Gemini에게 "최신 모델 찾아줘"를 시켰지만, 결과가 부정확하고 실제 라우팅에 반영되지도 않았다. 제거.

## 이 기획서의 용도

이 문서는 LLMTrio를 처음부터 다시 만들 때 참조할 기획서다. 현재 코드의 구현 세부사항이 아니라 "왜 이걸 만들고, 뭘 해야 하고, 뭘 하지 말아야 하는지"를 담았다.

코드가 전부 사라져도 이 문서가 있으면 같은 방향으로 다시 만들 수 있다.
